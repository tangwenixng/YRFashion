# MVP 详细设计：部署与运维

版本：v0.3  
日期：2026-03-14

## 1. 文档定位

本文承接 [04-MVP架构设计-Python后端版.md](./04-MVP架构设计-Python后端版.md)，聚焦一期部署形态、环境划分、上线准备和基础运维要求。

## 2. 推荐部署形态

一期推荐：

- 1 个微信小程序前端
- 1 个 Web 管理后台前端
- 1 个 FastAPI 云托管服务
- 1 个托管 MySQL 实例
- 1 个云存储桶

配套入口：

- 小程序调用正式 API 域名
- 后台通过正式 Admin 域名访问
- 图片通过云存储访问域名或 CDN 域名访问

## 3. CloudBase 侧建议

建议将生产形态直接建立在以下能力上：

- CloudBase 环境
- 云托管承载 FastAPI
- 静态网站托管承载后台前端
- HTTP 访问服务或自定义域名暴露 API

说明：

- 开发联调阶段可先用默认域名或本地联调
- 生产环境不建议长期依赖默认域名作为正式入口

## 4. 云环境与服务建议

一期可采用：

- 1 个 `prod` 环境
- 1 个 FastAPI 云托管服务
- 1 个后台静态网站托管站点
- 1 个托管 MySQL 实例
- 1 个云存储桶

如果预算优先：

- `dev` 使用较低规格
- 云托管最小实例数可从 `0` 起步
- 非核心环境数据库可采用更低配置

如果稳定性优先：

- 生产环境为 FastAPI 保留最小实例数
- 生产数据库避免自动休眠影响首请求时延
- 图片访问接入 CDN 或媒体加速域名

## 5. 环境划分

至少保留两个环境：

- `dev`：开发联调
- `prod`：正式环境

条件允许时增加：

- `staging`：提审前验收环境

建议做法：

- 不同环境使用独立环境变量
- 数据库、存储路径或存储桶按环境隔离
- 小程序、后台和后端的环境配置统一管理
- 不让开发环境直接共用生产数据库和生产存储

## 6. 域名与备案

正式上线前需要把“小程序备案”和“自定义域名备案”拆开规划。

关键判断：

- 使用 CloudBase / 云托管，不等于免除小程序备案。
- 如果存在正式 Web 后台、浏览器访问 API 或长期运营诉求，自定义域名仍应尽早备案。
- 本项目建议至少准备 `admin.xxx.com` 和 `api.xxx.com` 两个正式域名。
- 备案不应拖到最后，应与账号注册、环境搭建并行推进。

## 7. 上线入口建议

建议将正式入口固定为：

- 小程序正式版 -> 正式 API 域名
- 后台 Web -> 备案后的正式域名
- 图片资源 -> 云存储域名或 CDN 域名

这样可以减少从默认域名、临时地址、本地联调地址切换到正式域名时的返工。

## 8. 非功能设计

## 8.1 安全

- 全站 HTTPS
- 后台强密码策略
- 图片上传校验 MIME、大小、后缀
- 文件名使用 UUID，避免覆盖和路径穿透
- 留言内容长度限制与敏感词过滤预留
- 登录和留言接口限流
- 微信密钥、数据库连接、存储凭证通过环境变量或平台密钥管理
- 管理后台域名与 API 域名处理好跨域、鉴权和来源控制

## 8.2 可观测性

- 应用日志区分访问日志和错误日志
- 每个请求生成 `request_id`
- 记录关键后台操作日志，例如商品编辑、上下架、回复留言
- 接入 CloudBase 平台日志、监控指标和告警

## 8.3 备份

- 托管 MySQL 开启自动备份、快照或定期导出
- 云存储配置生命周期管理
- 重要配置支持导出
- 关键后台数据可定期导出，用于审计或恢复

## 8.4 性能

一期目标不是高并发，但仍建议：

- 商品列表接口分页
- 首页与商品列表增加短时缓存
- 图片提供压缩图和缩略图
- 图片访问优先走云存储访问域名或 CDN
- 如冷启动明显影响体验，提高最小实例数或增加保活策略

## 9. 演进原则

当前架构可以平滑演进到二期，但不建议过早引入复杂基础设施。

原则如下：

- 先保持单体应用，优先扩表和扩接口
- 消息通知、订单支付明显复杂后，再考虑拆模块
- Redis、任务队列、搜索服务都在明确需要时再引入

## 10. 推荐实施顺序

1. 确认页面清单和后台清单。
2. 输出数据库表结构和接口草案。
3. 初始化小程序、后台 Web、FastAPI 三个工程。
4. 搭建 `dev` 环境，准备 MySQL、云存储和基础环境变量。
5. 打通登录、商品、留言、后台商品管理主链路。
6. 接入图片上传、静态托管、域名和环境配置。
7. 并行推进小程序备案、域名备案和正式发布前验收。
