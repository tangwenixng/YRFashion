# 后台商品编辑弹框排版优化方案

## 1. 背景

当前后台商品管理页中的“新增 / 编辑”弹框已具备完整功能，支持：

- 基础信息填写
- 分类与标签设置
- 状态与排序值设置
- 多图上传
- 图片排序
- 封面设置
- 图片删除

对应实现位于：

- `frontend/src/views/ProductsView.vue`

从当前交互流程看，功能链路完整，但弹框内部的布局组织还不够稳定。用户在编辑内容时，需要自己在较长表单里寻找重点，尤其在图片较多时，弹框会明显失去重心。

本方案目标是：在不改变现有功能能力的前提下，优化弹框内部的信息结构、视觉节奏和操作收口，让编辑流程更清晰、更轻松。

## 2. 当前布局问题

### 2.1 信息分组不明显

当前弹框中：

- 商品名称
- 描述
- 分类
- 标签
- 状态
- 排序值
- 图片上传
- 图片排序与封面

这些内容虽然顺序合理，但视觉上几乎是连续堆叠的，缺少明确的分区提示。

结果是：

- 用户需要自己判断“当前在填什么”
- 上半部分基础信息没有形成一个完整区块
- 下半部分图片管理区域虽然更重，但没有被突出为独立模块

### 2.2 视觉重心下沉

图片管理区域包含：

- 上传拖拽区
- 图片卡片列表
- 排序提示
- 封面操作
- 删除操作

这一块体积远大于上方基础表单，导致整个弹框视觉重心明显落在下半部分。打开弹框后，用户的注意力会被图片卡片直接吸走，而不是先完成基础信息确认。

### 2.3 表单区节奏偏松散

当前表单存在以下问题：

- `描述` 区域高度偏高
- `分类 / 标签` 和 `状态 / 排序值` 两行控件形态不统一
- `排序值` 的信息权重偏低，但和 `状态` 处于同一级视觉层

这会导致上半部分看似留白充足，但信息组织没有形成明确节奏。

### 2.4 上传区与管理区层级不够清晰

目前“展示图片”和“图片顺序与封面”都放在同一视觉容器内，且风格接近。

问题在于：

- 用户第一眼不容易区分“这里是上传入口”还是“这里是已上传图片管理”
- 上传动作和图片管理动作混在一个大区块里
- 内容多时，层级边界会变弱

### 2.5 图片卡片信息偏杂

单张图片卡片目前包含：

- 上传状态
- 排序说明
- 大图预览
- 文件名
- 封面标记
- 待上传标记
- 设为封面
- 删除

这些信息都有效，但并不是同一层级。

当前问题是：

- 文件名太抢眼
- 操作按钮区视觉比图片状态更重
- “哪张是封面”“当前顺序是什么”这两个高频信息没有成为最先被感知的内容

### 2.6 底部操作收口偏弱

弹框底部目前只有：

- 取消
- 保存商品

功能上足够，但视觉收口不强。

对于这种带较长编辑流程的弹框，底部操作区应承担两个任务：

- 告诉用户“编辑快结束了”
- 明确主按钮是 `保存`

当前页脚更像普通默认 footer，没有形成明显的流程终点。

## 3. 优化目标

### 3.1 编辑目标

- 用户打开弹框后，能快速分辨“先填基础信息，再处理图片”
- 用户能在图片较多时快速判断封面、顺序和删除对象
- 用户在保存前不需要来回扫视整个弹框确认内容

### 3.2 布局目标

- 上半部分紧凑但不拥挤
- 下半部分图片区层级清楚
- 主操作明确，次操作克制
- 弹框即使内容多，也保持结构稳定

## 4. 推荐结构方案

建议将弹框明确拆成两个主区域：

1. 基础信息
2. 图片管理

建议在每个区域顶部增加区块标题和简短说明，让用户清楚自己当前正在处理哪一类内容。

## 4.1 基础信息区

建议内容包含：

- 商品名称
- 描述
- 分类
- 标签
- 状态
- 排序值

推荐布局：

- 第一行：商品名称
- 第二行：描述
- 第三行：分类 / 标签
- 第四行：状态 / 排序值

设计原则：

- 名称最重要，应放在最上方并占整行
- 描述是补充信息，保留但不需要过高
- 分类、标签、状态、排序值做成规则的两列结构

## 4.2 基础信息区优化建议

### 商品名称

- 保持整行展示
- 作为第一视觉焦点
- 输入框高度与其他控件统一

### 描述

- 建议将 textarea 高度从当前偏高状态收紧
- 优先控制在 `3 - 4 行`
- 说明型文本不宜过高，否则会挤压后续关键字段位置

### 分类 / 标签 / 状态 / 排序值

建议采用更稳定的双列栅格：

- 左列：分类、状态
- 右列：标签、排序值

约束建议：

- 控件高度统一
- `排序值` 区块增加简短说明，例如“值越小越靠前”
- `排序值` 不应在视觉上和主业务字段竞争注意力

## 4.3 图片管理区

图片管理区建议再拆成两个子层：

1. 图片上传区
2. 已上传图片管理区

这两个区域虽然同属“图片管理”，但用户目标不同：

- 上传区关注“新增图片”
- 管理区关注“顺序 / 封面 / 删除”

建议在视觉上做明确区隔，不要让两者看起来像同一种卡片。

## 4.4 图片上传区建议

当前上传区使用拖拽上传是合理的，但建议优化层级：

- 作为图片区第一个子模块
- 标题更聚焦，例如：`上传图片`
- 说明文案更短，只强调格式和大小限制

建议表达：

- 标题：`上传图片`
- 副文案：`支持 JPG / PNG / WEBP，单张不超过 5MB`

拖拽区本身应更像“动作入口”，而不是和下面的图片区块抢视觉焦点。

## 4.5 已上传图片管理区建议

建议单独命名为：

- `图片顺序与封面`

并增加一句简短说明：

- `拖动卡片调整顺序，封面将用于列表和首页预览`

与上传区的区分原则：

- 上传区强调“新增”
- 管理区强调“整理”

## 4.6 图片卡片优化建议

建议将单张图片卡片拆成三个层次：

1. 状态层
2. 图片层
3. 操作层

### 状态层

建议放在卡片顶部，展示：

- 当前顺序
- 是否封面
- 是否待上传

目标是让用户先看“这张图在整个序列里是什么角色”。

### 图片层

- 维持大图预览
- 预览区域是卡片的主体
- 保证不同图片尺寸下显示稳定

### 操作层

建议只保留两个核心动作：

- 设为封面
- 删除

按钮风格建议：

- `设为封面`：普通描边按钮
- `删除`：危险按钮，但弱化到不抢主视觉

## 4.7 文件名展示建议

文件名属于低优先级信息。

当前直接放在卡片主体里，容易造成以下问题：

- 文字过长，破坏卡片节奏
- 拉高卡片文本区
- 抢走对“封面 / 顺序 / 操作”的注意力

建议改为以下任一方案：

- 默认只展示单行截断
- hover 时通过 tooltip 查看完整文件名
- 或直接降到卡片底部最弱层级

优先建议：

- 卡片内保留单行截断
- 完整文件名通过 `title` 或 tooltip 查看

## 4.8 弹框宽度与比例建议

当前弹框宽度为 `920px`，在图片较多时仍然显得略紧。

建议方向：

- 若继续维持双列图片卡片布局，可考虑适度增加弹框宽度
- 若保持当前宽度，则应减少卡片内部文本量

核心原则不是单纯“加宽”，而是让：

- 上半部分表单不压缩
- 下半部分图片卡片不过度拥挤

## 4.9 页脚操作区建议

建议强化底部 footer 的收口感：

- `取消` 保持次要按钮
- `保存商品` 保持主按钮
- 页脚与内容区之间增加更明显的分隔

可选增强：

- 新增一段弱提示，例如 `保存后将同步更新列表展示`

这样用户在长编辑流程结束时，能更自然地完成“确认保存”。

## 5. 推荐结构草图

建议结构如下：

```text
编辑商品

[基础信息]
商品名称
描述
分类        标签
状态        排序值

[图片管理]
上传图片
[拖拽上传区]

图片顺序与封面
[图片卡片] [图片卡片]
[图片卡片] [图片卡片]

取消                    保存商品
```

## 6. 推荐改造顺序

如果进入实现阶段，建议按以下顺序修改：

1. 先拆出“基础信息”和“图片管理”两大分区
2. 再压缩描述高度、统一基础字段控件节奏
3. 再分离上传区与图片管理区的视觉层级
4. 最后收敛图片卡片中的文件名、标签和按钮样式

这样可以在每一步都获得明显收益，并避免一次改动过大。

## 7. 对应实现建议

建议优先修改以下内容：

- `ProductsView.vue` 中编辑弹框的模板结构
- `ProductsView.vue` 中 `editor-grid`、`inline-grid`、`editor-media-panel` 相关样式
- 图片卡片中的 `image-card-meta`、`image-card-tags`、`image-card-actions`
- 弹框 footer 的间距和分隔层次

## 8. 验收标准

优化后的编辑弹框应满足：

- 用户打开后能快速识别基础信息区和图片管理区
- 基础字段排列更整齐，不再显得松散
- 图片上传区和已上传图片区职责清楚
- 图片卡片的封面状态和顺序更容易识别
- 长文件名不再明显干扰图片管理操作
- 底部 `保存商品` 成为明确的流程收口点

## 9. 结论

当前编辑弹框的问题不在于功能不足，而在于排版还没有把“编辑流程”组织出来。

这次优化的核心目标是：

- 让基础信息先被清楚填写
- 让图片管理更容易判断和整理
- 让用户在保存前不需要反复搜索页面重点

这是一项适合逐步实施的低风险改版，适合作为后台商品编辑体验优化的下一阶段依据。
