# 二期功能规划

版本：v0.1  
日期：2026-03-15
更新：2026-03-27（基于当前代码再次核对）

> 状态说明：以下标题后的 `[已完成]`、`[部分完成]`、`[未完成]`、`[文档已过时]` 为截至 2026-03-27 与当前系统实现的对照结果。本文原有规划顺序保留，但其中 P1 已大多落地，阅读时需要结合状态标记理解。

## 1. 规划目标

当前版本已经完成第一版 MVP，核心链路是：

- 小程序端完成“展示 + 咨询 + 联系”
- 管理后台完成“内容维护 + 留言处理 + 店铺配置”

现阶段系统的主要问题不是“功能不能用”，而是：

- 小程序端找内容的效率偏低
- 用户发起咨询后缺少后续反馈闭环
- 后台在数据规模增长后运营效率有限
- 当前统计能力较弱，不利于判断哪些内容更有效

因此，二期不建议直接扩展为完整电商系统，而建议继续围绕“展示获客系统”升级，优先补齐更容易提升转化、更容易持续运营的能力。

## 2. 二期定位

二期系统定位为：

- 更容易浏览和发现内容的穿搭展示小程序
- 更完整的用户咨询与店主回复闭环
- 更适合持续运营的轻量后台系统
- 能支持基础数据判断和运营决策的内容管理平台

二期暂不建议纳入：

- 支付
- 购物车
- 订单
- 优惠券
- 复杂会员体系
- 多角色复杂权限模型

原因是这些能力会显著抬高数据模型复杂度、审核风险和运维成本，不适合作为当前阶段的优先投入方向。

## 3. 排优原则

二期功能优先级按以下原则排序：

- 高价值：能明显提升用户转化、咨询率或运营效率
- 中成本：可在现有前后端结构上连续扩展，不需要重构主架构
- 低风险：不显著增加审核、交易、履约、售后或复杂权限风险

## 4. 建议优先级

### 4.1 P1：优先落地 `[已完成]`

这一层功能最符合“高价值 / 中成本 / 低风险”，建议作为二期第一批实施内容。

#### 4.1.1 小程序商品分类筛选 `[已完成]`

目标：

- 让用户更快找到想看的内容
- 降低列表信息变多后的浏览成本

建议内容：

- 商品列表页增加分类筛选
- 支持按后台启用分类展示
- 支持默认“全部”分类

价值判断：

- 价值：高
- 成本：中
- 风险：低

原因：

- 后台已具备分类能力，前后端已有基础数据结构
- 该功能能直接提升列表页可用性

#### 4.1.2 小程序商品搜索 `[已完成]`

目标：

- 支持用户按名称、关键词快速定位内容

建议内容：

- 商品列表页增加搜索框
- 支持按名称、标签做模糊搜索
- 搜索结果为空时提供空态提示

价值判断：

- 价值：高
- 成本：中
- 风险：低

原因：

- 搜索是最直接的内容发现能力
- 对现有商品数据模型改动有限

#### 4.1.3 小程序列表分页与下拉刷新 `[已完成]`

目标：

- 解决商品数量增长后的加载效率问题
- 优化列表浏览体验

建议内容：

- 商品列表接口改为分页返回
- 前端支持上拉加载更多
- 前端支持下拉刷新

价值判断：

- 价值：高
- 成本：中
- 风险：低

原因：

- 文档中已经将分页列为可选增强
- 属于标准能力，技术实现稳定

#### 4.1.4 小程序留言历史与回复展示 `[已完成]`

目标：

- 把“发起咨询”升级为“可跟进的咨询闭环”

建议内容：

- 用户可查看自己的留言历史
- 展示店主是否已查看
- 展示店主回复内容
- 支持按商品维度查看历史咨询

价值判断：

- 价值：高
- 成本：中
- 风险：低

原因：

- 后台已经有留言和回复能力，主要缺少小程序展示端
- 这项功能能明显提升用户感知，不再是“留言后无反馈”

#### 4.1.5 后台列表分页、筛选与批量操作 `[已完成]`

目标：

- 提升后台在内容和消息增长后的运营效率

建议内容：

- 商品列表增加分页、状态筛选、分类筛选、关键词筛选
- 留言列表增加分页、状态筛选、商品筛选
- 用户列表增加分页、关键词筛选
- 增加批量已读、批量归档、批量上下架等操作

价值判断：

- 价值：高
- 成本：中
- 风险：低

原因：

- 当前后台已经完成基础 CRUD，但运营能力偏轻
- 这类增强不改变核心业务模型，收益稳定

### 4.2 P2：第二批落地 `[部分完成]`

这一层功能仍然值得做，但优先级略低于 P1，适合在第一批增强完成后推进。

#### 4.2.1 后台账号管理与密码修改 `[文档已过时]` `[已完成]`

目标：

- 为多人协作和正式运营做准备

建议内容：

- 后台账号列表
- 新建后台账号
- 修改密码
- 禁用账号

价值判断：

- 价值：中高
- 成本：中
- 风险：中低

原因：

- 当前只有默认管理员，单人可用，但不适合后续协作
- 会涉及鉴权和后台管理规则，复杂度高于普通列表功能

#### 4.2.2 咨询数据统计 `[部分完成]`

目标：

- 让后台知道哪些内容更容易带来咨询

建议内容：

- 商品咨询次数排行
- 最近 7 天 / 30 天咨询趋势
- 未读留言趋势
- 用户来源基础统计

价值判断：

- 价值：中高
- 成本：中
- 风险：低

原因：

- 这类能力有助于内容运营决策
- 数据来源基本都在现有库内，可先做轻量统计

#### 4.2.3 小程序推荐内容增强 `[部分完成]`

目标：

- 提升首页和详情页的继续浏览率

建议内容：

- 首页增加“本周推荐”规则配置
- 商品详情页增加相关推荐
- 基于分类或标签做简单推荐

价值判断：

- 价值：中高
- 成本：中
- 风险：低

原因：

- 能提升页面停留和内容消费深度
- 可先使用简单规则，不必上复杂推荐算法

#### 4.2.4 首页与列表缓存优化 `[未完成]`

目标：

- 降低重复请求
- 提升小程序打开速度和稳定性

建议内容：

- 首页短时缓存
- 商品列表短时缓存
- 缓存失效后自动刷新

价值判断：

- 价值：中
- 成本：中
- 风险：低

原因：

- 用户感知明显，但不如搜索、筛选、咨询闭环直接
- 实现方式成熟，适合作为性能优化项

### 4.3 P3：暂缓实施 `[未完成]`

以下能力可以保留在更后续阶段，不建议作为当前二期的主要投入方向。

#### 4.3.1 支付、购物车、订单 `[未完成]`

暂缓原因：

- 会把系统从“展示获客”推向“交易履约”
- 需要重做商品、库存、订单、售后、支付状态等模型
- 风险和实施成本都明显上升

#### 4.3.2 优惠券、会员体系 `[未完成]`

暂缓原因：

- 依赖交易或至少依赖完整用户成长体系
- 当前阶段业务闭环仍以咨询转化为主

#### 4.3.3 复杂预约体系 `[未完成]`

暂缓原因：

- 需要额外设计时段、资源占用、确认流程和违约处理
- 当前联系页与留言能力已能满足轻量预约场景

#### 4.3.4 多角色复杂权限模型 `[未完成]`

暂缓原因：

- 当前更适合先做“多账号 + 基础权限”
- 复杂 RBAC 会引入较高维护成本

## 5. 推荐实施顺序

当前对照说明：

- 原建议顺序本身仍然成立，但前两步中的多项能力已落地，因此本节部分内容已可视为 `[文档已过时]`
- 现阶段更值得优先补的是统计深化、推荐规则配置、缓存优化

建议按以下顺序推进二期：

1. 小程序分类筛选
2. 小程序搜索
3. 小程序分页与下拉刷新
4. 小程序留言历史与回复展示
5. 后台列表分页、筛选与批量操作
6. 后台账号管理与密码修改
7. 咨询数据统计
8. 推荐内容增强
9. 缓存与性能优化

这个顺序的核心思路是：

- 先解决“用户找不到内容”的问题
- 再解决“咨询没有回路”的问题
- 然后解决“后台运营效率不足”的问题
- 最后补“统计和优化能力”

## 6. 建议拆分为三个迭代

### 6.1 迭代一：提升小程序可用性 `[已完成]`

建议范围：

- 分类筛选
- 商品搜索
- 列表分页
- 下拉刷新

目标：

- 先把小程序内容发现能力做起来

### 6.2 迭代二：补齐咨询闭环 `[部分完成]`

建议范围：

- 留言历史
- 回复展示
- 咨询状态展示
- 相关推荐

目标：

- 提高咨询体验与后续跟进感知

### 6.3 迭代三：增强后台运营能力 `[部分完成]`

建议范围：

- 后台分页与筛选
- 批量操作
- 后台账号管理
- 统计看板增强

目标：

- 让系统从“能用”升级到“适合持续运营”

## 7. 一句话结论

截至 2026-03-27，本文件中原先定义的 P1 已基本完成；当前真正仍需推进的重点，主要是 `[部分完成]` 的统计与推荐增强，以及 `[未完成]` 的缓存优化。

二期最合理的方向，不是把系统直接做成商城，而是继续把它做深成一个更强的“穿搭展示 + 咨询转化 + 后台运营”系统。优先补搜索、筛选、分页、咨询闭环和后台运营增强，这些功能最符合当前阶段的高价值、中成本、低风险原则。
