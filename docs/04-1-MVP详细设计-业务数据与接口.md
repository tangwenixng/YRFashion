## 2. 系统分层

建议后端保持标准单体分层，而不是把逻辑堆在路由层。

```text
API 层
  - 小程序接口
  - 后台管理接口

应用服务层
  - 登录服务
  - 商品服务
  - 留言服务
  - 用户服务
  - 媒体服务
  - 店铺配置服务

领域模型层
  - Product
  - ProductImage
  - MiniappUser
  - Message
  - AdminUser
  - Category
  - ShopSetting

基础设施层
  - MySQL 仓储
  - 云存储适配器
  - 微信登录适配器
  - 通知适配器
  - 日志与配置
```

### API 层职责

- 参数校验
- 权限校验
- 统一响应结构
- 错误码输出

### 应用服务层职责

- 承接登录、商品发布、留言回复等具体用例
- 编排仓储、存储、微信接口等依赖

### 基础设施层职责

- 对接托管 MySQL
- 对接云存储
- 对接微信 `code2Session`
- 为后续通知能力预留接入点

## 3. 功能模块设计

## 3.1 小程序用户模块

职责：

- 调用 `wx.login()`
- 将 `code` 传给后端
- 后端换取 `openid`
- 建立本站用户记录与登录态

设计建议：

- 以 `openid` 作为唯一外部标识
- 昵称、头像仅作为补充字段
- 首次访问自动建档，记录最近访问时间

## 3.2 后台账号模块

职责：

- 店主登录
- 后台接口鉴权

设计建议：

- 一期只做单管理员或少量后台账号
- 使用账号密码登录
- 密码强哈希存储，例如 `bcrypt`
- 不引入复杂 RBAC，只保留 `admin` 单角色

## 3.3 商品模块

职责：

- 商品新增与编辑
- 上下架
- 分类/标签维护
- 列表展示
- 排序控制

设计建议：

- 商品主表与图片表分离
- 排序字段统一使用 `sort_order`
- 小程序端只返回已上架商品

## 3.4 图片模块

职责：

- 图片上传
- 图片元数据保存
- 封面图指定
- 图片排序

一期实现建议：

- 管理后台先把图片传给后端
- 后端完成文件校验后上传云存储
- 返回访问地址并写入数据库

后续优化建议：

- 文件存储接口保持抽象
- 图片量变大后再演进为前端直传 + 后端签发凭证

## 3.5 留言模块

职责：

- 用户对商品提交留言
- 店主查看和回复留言
- 维护 `unread` / `read` / `replied` 状态

设计建议：

- 留言与商品强关联
- 一期采用单条留言 + 单次回复模型
- 多轮咨询留到二期再升级为会话模型

## 3.6 用户记录模块

职责：

- 记录小程序用户基础资料
- 记录首次访问与最近访问时间

一期不做：

- 用户画像
- 积分
- 会员等级
- 行为埋点平台

## 3.7 店铺配置模块

职责：

- 联系方式
- 门店地址
- 营业时间
- 店铺介绍
- 首页展示文案

## 4. 核心业务流程

## 4.1 小程序登录流程

```text
1. 小程序调用 wx.login() 获取 code
2. 小程序把 code 发送给后端
3. 后端调用微信 code2Session
4. 后端拿到 openid / session_key
5. 若用户不存在则创建用户
6. 后端签发本系统 access token
7. 小程序后续带 token 调用业务接口
```

补充约束：

- `session_key` 不长期暴露给前端
- 后端只保留业务需要的信息

## 4.2 商品发布流程

```text
1. 管理员登录后台
2. 新建商品基本信息
3. 上传商品图片
4. 后端上传图片到云存储并记录图片元数据
5. 设置封面图、排序、上下架状态
6. 小程序商品列表按已上架 + sort_order 展示
```

## 4.3 留言处理流程

```text
1. 用户进入商品详情页
2. 提交留言内容
3. 后端写入 messages 表，状态为 unread
4. 后台首页显示未读数
5. 店主查看留言并标记已读或直接回复
6. 回复后状态改为 replied
```

## 5. 数据库设计草案

以下表结构按 MySQL 关系型数据库设计，不对应 CloudBase 原生文档型数据库的数据建模方式。

## 5.1 `admin_users`

- `id`
- `username`
- `password_hash`
- `display_name`
- `status`
- `last_login_at`
- `created_at`
- `updated_at`

## 5.2 `miniapp_users`

- `id`
- `openid`
- `unionid` 可空
- `nickname` 可空
- `avatar_url` 可空
- `first_visit_at`
- `last_visit_at`
- `created_at`
- `updated_at`

约束建议：

- `openid` 唯一索引

## 5.3 `categories`

- `id`
- `name`
- `sort_order`
- `status`
- `created_at`
- `updated_at`

说明：

- 如果一期分类很轻，也可以暂时不单独建表，先使用标签字段

## 5.4 `products`

- `id`
- `name`
- `category_id` 可空
- `cover_image_id` 可空
- `description`
- `tags_json`
- `sort_order`
- `status`，如 `draft` / `published` / `archived`
- `created_at`
- `updated_at`

索引建议：

- `status + sort_order`
- `category_id + status + sort_order`

## 5.5 `product_images`

- `id`
- `product_id`
- `storage_type`，一期固定为 `cloud`
- `storage_path`
- `image_url`
- `original_name`
- `width` 可空
- `height` 可空
- `sort_order`
- `is_cover`
- `created_at`

说明：

- `storage_path` 表示云存储中的对象路径或 Key
- `image_url` 表示对外访问地址

## 5.6 `messages`

- `id`
- `product_id`
- `miniapp_user_id`
- `content`
- `status`
- `reply_content` 可空
- `reply_at` 可空
- `read_at` 可空
- `created_at`
- `updated_at`

索引建议：

- `product_id + created_at`
- `status + created_at`
- `miniapp_user_id + created_at`

## 5.7 `shop_settings`

- `id`
- `shop_name`
- `shop_intro`
- `contact_phone`
- `wechat_id`
- `address`
- `business_hours`
- `hero_text`
- `created_at`
- `updated_at`

## 6. 接口边界设计

## 6.1 小程序接口

建议保留以下主链路接口：

- `POST /api/miniapp/auth/login`
- `GET /api/miniapp/products`
- `GET /api/miniapp/products/{id}`
- `POST /api/miniapp/messages`
- `GET /api/miniapp/shop/settings`

约束：

- 商品列表只返回已上架数据
- 留言提交需要登录态
- 小程序端不承载店主处理逻辑

## 6.2 后台接口

建议保留以下主链路接口：

- `POST /api/admin/auth/login`
- `GET /api/admin/products`
- `POST /api/admin/products`
- `PUT /api/admin/products/{id}`
- `POST /api/admin/products/{id}/images`
- `PUT /api/admin/products/{id}/status`
- `GET /api/admin/messages`
- `PUT /api/admin/messages/{id}/read`
- `POST /api/admin/messages/{id}/reply`
- `GET /api/admin/shop/settings`
- `PUT /api/admin/shop/settings`

约束：

- 后台接口统一走管理员鉴权
- 图片上传单独接口处理，避免把二进制文件直接混入商品更新接口

## 7. 认证与权限

## 7.1 小程序端认证

- 登录来源：`wx.login()` + 后端换取 `openid`
- 后端签发本站 `access token`
- 业务接口以本站 token 为准，不直接信任前端传来的微信身份字段

## 7.2 后台认证

- 账号密码登录
- 登录成功后签发后台 token
- 一期只保留 `admin` 角色，不展开复杂权限模型

## 8. 二期演进边界

以下能力不在一期落地，但当前设计应为它们保留演进空间：

- 收藏/预约
- 订阅通知
- 营销活动
- 订单
- 支付

演进原则：

- 先继续保持单体
- 明确出现复杂异步任务、支付链路或高频查询后，再引入 Redis、队列或独立模块
