# 部署方案取舍记录

日期：2026-03-14

## 当前判断

项目后端后续大概率会从【云托管】回切到自有服务器部署，采用自己更熟悉也更可控的方案：

- Python 后端
- MySQL
- 本地存储或 MinIO
- `docker compose`
- 自有域名 + HTTPS

这不是能力问题，而是平台方案与当前项目阶段、个人运维习惯、排障方式不匹配。

## 在云托管上遇到的主要问题

这次实践后，确认【云托管】虽然降低了一部分基础设施门槛，但也带来了明显的管理成本：

1. 管理不便，尤其是日志查看和排障体验不够直接。
2. `docker compose` 编排不方便，不符合已有的部署和维护习惯。
3. 使用 MySQL 的方式不够顺手，整体体验不如自管环境。
4. 平台提供的公网域名更适合测试，正式环境最终仍然需要自有域名。

原本预期是“用了云托管就可以少管很多事”，但实际效果是额外增加了不少适配、调试和运维上的麻烦。

## 回切到自有服务器的原因

相比之下，自有服务器方案更符合当前项目的实际需求：

- 对日志、容器、网络、数据和文件存储有完整控制权。
- 部署方式和排障手法都已经非常熟练。
- `docker compose + MySQL + MinIO/本地存储` 的组合更稳定、更可预期。
- 正式上线本来也需要配置自有域名和 HTTPS，这部分并不会因为使用云托管而真正消失。

因此，从长期维护、效率和可控性来看，回到原来的方案更合理。

## 当前结论

后端部署方案暂定收口为：

- 应用：Python
- 数据库：MySQL
- 文件存储：本地存储或 MinIO
- 编排方式：`docker compose`
- 访问入口：自有域名 + HTTPS

云托管这次尝试并非完全没有价值。它帮助明确了哪些地方会增加系统复杂度，也验证了当前项目更适合走“自有服务器、自己掌控”的路线。

## 额外说明

值得庆幸的是，代码里原有的本地存储逻辑没有被移除。这意味着后续如果回切部署方案，不需要为存储层做大幅返工，迁移成本会低很多。
