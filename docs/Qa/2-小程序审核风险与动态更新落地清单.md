# 小程序审核风险与动态更新落地清单

这份文档用于记录结合当前项目得出的结论、风险边界和开发任务清单，不混入原始对话记录。

---

# 结合当前项目的落地结论

结合当前项目实现，最合适的方向不是做“任意动态页面引擎”，而是：

```text
固定页面骨架 + 后台动态内容 + 内容安全审核 + 发布态控制
```

原因：

* 当前项目已经具备基础的后台动态更新能力
* 首页 banner、店铺介绍、联系方式、推荐商品都可以走服务端返回
* 商品列表和商品详情本身也是数据库驱动
* 这样可以减少频繁提审，但不会踩“动态下发未审核功能”的红线

当前项目更适合的原则：

* 小程序前端负责展示，不负责承载经常变化的运营内容
* 所有可变内容尽量走后台配置和数据库
* 所有动态内容都必须限制在已审核通过的业务范围内
* 不能把“动态更新内容”做成“动态加载代码”或“远程下发任意页面”

# 当前项目的主要审核风险

## 1. 动态更新不等于可以绕过审核

后端动态返回数据本身没有问题，但如果通过后端动态增加了新的核心功能、服务类目、主流程页面，仍然可能触发审核风险。

安全边界应该是：

* 可以动态改：banner、文案、商品、推荐位、联系方式
* 不能偷偷动态改：新业务功能、新类目能力、新页面类型、新交易链路、新导流能力

## 2. 不能把动态布局做成动态代码

如果只是前端预埋一组固定组件，由后端控制顺序、显隐、数量，这通常是可控的。

但如果后面做成下面这些，就有风险：

* 服务端下发 HTML
* 服务端下发 JS 逻辑
* `web-view` 承载核心主流程
* 运行时加载未审核模板

更稳的方式是：

* 组件类型白名单
* 前端内置组件
* 后端只下发配置和数据

## 3. 当前项目已经存在 UGC，必须补内容安全

当前项目有用户留言、昵称更新、头像上传，这些都属于审核风险点。

至少要补：

* 留言内容安全校验
* 昵称内容安全校验
* 头像图片安全校验或待审机制
* 后台 banner 图、商品图的审核或发布前人工复核

## 4. 联系方式展示要避免导流话术

展示电话、地址、营业时间通常风险较低。

但如果页面文案出现类似：

* 加微信下单
* 加微信领券
* 扫码进群
* 私聊获取资源

就会明显提高审核风险。

更稳妥的做法：

* 联系页主展示电话、地址、营业时间
* 微信号仅作为补充联系方式
* 不要配返利、裂变、加群、私下交易等文案

# 当前项目的开发任务清单

下面按优先级整理成可执行任务。

## P0 合规兜底

### 1. 接入文本内容安全校验

目标：

* 对小程序留言内容做文本安全校验
* 对小程序昵称更新做文本安全校验

涉及范围：

* `backend/src/backend/api/routes/miniapp_messages.py`
* `backend/src/backend/api/routes/miniapp_auth.py`

验收标准：

* 留言提交前调用内容安全接口
* 昵称更新前调用内容安全接口
* 违规内容不落库
* 接口返回明确提示

### 2. 接入图片内容安全校验或待审机制

目标：

* 对头像上传建立安全校验
* 对 banner 图、商品图建立待审或人工复核机制

涉及范围：

* `backend/src/backend/api/routes/miniapp_auth.py`
* `backend/src/backend/api/routes/admin_products.py`
* `backend/src/backend/api/routes/admin_settings.py`

验收标准：

* 用户头像上传后不会直接无审展示
* 后台图片可标记为待审
* 审核未通过的图片不会出现在小程序端

### 3. 收紧联系方式与导流文案策略

目标：

* 保留必要联系方式
* 避免诱导添加、诱导私下交易的展示方式

涉及范围：

* `backend/src/backend/api/routes/miniapp_storefront.py`
* `frontend/src/views/SettingsView.vue`

验收标准：

* 小程序联系页不出现导流型文案
* 后台配置页有填写提示

### 4. 补内部运营合规文档

目标：

* 形成统一规则，避免后续运营内容越界

验收标准：

* 明确哪些内容可动态更新
* 明确哪些改动必须重新提审
* 明确图片、文案、联系方式的合规边界

## P1 动态更新能力标准化

### 1. 给店铺设置增加草稿与发布态

目标：

* 解决“后台一改立刻生效”的风险

涉及范围：

* `backend/src/backend/models/shop_setting.py`
* `backend/src/backend/api/routes/admin_settings.py`

验收标准：

* 支持保存草稿
* 支持显式发布
* 小程序端只读取已发布版本

### 2. 给 banner 建独立结构

目标：

* 不再只保存 URL 数组
* 为后续排序、审核、启用状态、标题、跳转预留能力

建议能力：

* 图片地址
* 排序
* 启用状态
* 审核状态
* 可选标题
* 可选跳转配置

验收标准：

* 小程序首页只读取已发布且启用的 banner
* banner 支持独立管理和排序

### 3. 首页模块做白名单配置

目标：

* 支持有限的模块顺序和显隐控制
* 不做任意页面引擎

建议白名单：

* `banner`
* `shop_intro`
* `featured_products`
* `contact_entry`

验收标准：

* 前端内置固定组件
* 后端只返回模块配置和数据
* 不支持远程模板、HTML、JS

### 4. 推荐商品改为后台显式配置

目标：

* 不再完全依赖排序默认截取
* 让运营能更稳定地维护首页推荐位

验收标准：

* 后台可设置推荐商品
* 小程序首页优先返回推荐位
* 无推荐时再回退默认排序逻辑

## P1 审核与追溯

### 1. 增加后台操作日志

目标：

* 记录谁修改了什么，便于追责和回滚

验收标准：

* 记录操作人、时间、对象、变更摘要
* 关键发布动作可追溯

### 2. 增加内容审核状态字段

目标：

* 对留言、头像、商品图、banner 建立统一审核状态

建议状态：

* `pending`
* `approved`
* `rejected`

验收标准：

* 未审核通过内容不会对外展示
* 后台可查询审核状态

### 3. 增加回滚能力

目标：

* 让首页配置和店铺设置可快速恢复

验收标准：

* 可以查看历史版本
* 可以一键回滚到上一个发布版本

## P1 后台前端配套

### 1. 后台设置页增加合规提示

目标：

* 把规则前置到录入界面，降低运营误操作

涉及范围：

* `frontend/src/views/SettingsView.vue`

验收标准：

* 字段旁有明确提示
* 提示覆盖导流文案、广告词、违规图片风险

### 2. 后台增加发布与预览分离

目标：

* 编辑不等于发布
* 发布前能看到小程序首页效果

验收标准：

* 支持预览
* 支持显式发布
* 非发布内容不影响线上

### 3. 后台增加审核失败反馈展示

目标：

* 当内容安全校验失败时，运营能知道如何处理

验收标准：

* 能看到失败原因或通用提示
* 不会只能看到模糊错误码

## P2 稳定性与风控增强

### 1. 敏感词本地预检

目标：

* 在调用微信安全接口前先挡住明显违规词

验收标准：

* 高频违规词可直接拦截
* 与微信官方安全校验互补

### 2. 提交频率限制

目标：

* 防刷内容
* 降低风控风险

验收标准：

* 留言和资料更新具备基础限流能力
* 高频重复提交会被拦截

### 3. 图片审核异步任务化

目标：

* 避免图片校验阻塞同步接口

验收标准：

* 上传后状态清晰
* 审核失败可重试

### 4. 监控与告警

目标：

* 让安全接口调用和审核失败情况可观测

验收标准：

* 内容安全接口失败率可统计
* 审核驳回数量可查看
* 发布异常可告警

---

# 建议实施顺序

建议按下面顺序推进：

1. 文本内容安全校验
2. 图片待审或图片安全校验
3. 店铺设置草稿与发布态
4. banner 独立化
5. 首页模块白名单配置
6. 操作日志与回滚
7. 限流、预检、监控

---

# 最小可上线版本

如果要用最小成本先落地，建议先完成这 4 项：

1. 留言和昵称接入文本安全校验
2. 头像上传增加待审或安全检测
3. 店铺设置改为草稿加发布
4. 后台设置页增加合规提示

这样可以先把最明显的审核风险降下来，同时保留后续继续扩展的空间。
