# 手机浏览器后台管理实施与验收说明

版本：v0.1  
日期：2026-04-17

## 1. 文档目的

本文用于给“手机小屏浏览器后台管理界面”一期提供一份可直接执行的实施/评审/验收基线，服务三个目标：

1. 把 `.omx/plans/prd-mobile-browser-admin.md` 的关键约束落到具体代码边界上。
2. 把 `.omx/plans/test-spec-mobile-browser-admin.md` 的核心验证点整理成开发联调与上线前手工回归清单。
3. 在不干扰桌面后台的前提下，为路由、鉴权、商品、留言等改动提供统一 review 检查项。

> 结论先行：本期应继续沿用 **单一 SPA + 显式 `/m` 路由命名空间**，复用现有后台 API、登录态和数据模型，不新建第二套前端构建产物，也不把 `/m` 误当成独立 history base。

## 2. 一期范围与非范围

### 2.1 一期必须覆盖

- `/m/login`：手机登录
- `/m/home`：手机首页/概览
- `/m/products`：商品列表
- `/m/products/create`：新增商品
- `/m/products/:id/edit`：商品编辑
- `/m/products/:id/images`：商品图片管理
- `/m/messages`：留言列表
- `/m/messages/:id`：留言详情/回复

### 2.2 一期不单独建设

- `/m/accounts`
- `/m/users`
- `/m/settings`
- iPad 专属布局
- 第二套移动端后端接口
- 独立移动端 SPA/独立部署产物

## 3. 实施边界（代码视角）

### 3.1 路由与入口边界

必须满足以下原则：

1. `/m` 是 **路由命名空间**，不是新的部署 base。
2. UA 自动分流只允许发生在 **中性入口**：`/`、`/login`、登录成功默认落点、401 回跳。
3. 用户一旦进入显式 URL（例如 `/products` 或 `/m/products`），默认保持当前路由族，不做静默改写。
4. 隐藏前缀部署必须继续兼容，例如：
   - `/login`
   - `/m/login`
   - `/gate/login`
   - `/gate/m/login`

建议涉及文件：

- `frontend/src/router/base.ts`
- `frontend/src/router/index.ts`
- `frontend/src/router/routes.ts`
- `frontend/src/router/routes.desktop.ts`
- `frontend/src/router/routes.mobile.ts`
- `frontend/src/router/device.ts` 或 `frontend/src/utils/deviceExperience.ts`

### 3.2 鉴权与会话恢复边界

必须复用现有后台鉴权与 API 模块，不另起一套 token 体系。

关键要求：

- 登录成功后根据当前体验族落到 `/dashboard` 或 `/m/home`
- 401 失效跳转保留当前路由族：
  - 桌面：`/login?redirect=<explicit-route>`
  - 手机：`/m/login?redirect=<explicit-route>`
- 退出登录时清除体验 override，恢复 `auto`
- 手机端与桌面端共用认证结果，但不能互相污染显式路由判断

建议涉及文件：

- `frontend/src/api/http.ts`
- `frontend/src/stores/auth.ts`
- `frontend/src/views/LoginView.vue`

### 3.3 手机壳层与页面边界

手机端不建议强行复用桌面表格与布局骨架；应尽量复用业务逻辑、重新组织展示层。

建议新增：

- `frontend/src/layouts/MobileAdminLayout.vue`
- `frontend/src/views/mobile/HomeView.vue`
- `frontend/src/views/mobile/ProductsView.vue`
- `frontend/src/views/mobile/ProductEditorView.vue`
- `frontend/src/views/mobile/ProductImagesView.vue`
- `frontend/src/views/mobile/MessagesView.vue`
- `frontend/src/views/mobile/MessageDetailView.vue`

移动页面原则：

- 单列卡片流
- 主操作就近可见
- 表单按分区或分步组织
- 列表优先“查看 + 快速操作”，减少桌面式宽表格依赖

## 4. 代码 Review 必查项

### 4.1 路由/基础设施

- [ ] `resolveAdminBasePath()` 或等价逻辑不会把 `/m` 识别成新的部署根
- [ ] `/gate/login` 与 `/gate/m/login` 在构建产物里都能正常解析
- [ ] neutral entry 才做 UA 分流，显式深链不被自动改写
- [ ] 路由 title、requiresAuth、guestOnly 元信息在手机/桌面两套路由中一致可维护

### 4.2 体验判定与 override

- [ ] override 至少支持 `auto / force-mobile / force-desktop`
- [ ] override 刷新后仍生效
- [ ] logout 会清除 override
- [ ] 未设置 override 时才使用 UA 自动判断
- [ ] 体验判定逻辑集中在一个模块中，避免散落在视图里重复判断

### 4.3 登录与 401 恢复

- [ ] 登录成功优先回到 `redirect`，没有 `redirect` 才走默认首页
- [ ] 401 恢复时保留当前显式 URL，避免手机页面被送回桌面登录页
- [ ] `/m/login` 与 `/login` 均能处理游客态和已登录态
- [ ] `http` 拦截器与路由守卫不会互相重复跳转，避免循环重定向

### 4.4 商品模块

- [ ] 手机商品列表支持搜索与状态筛选
- [ ] 卡片信息至少包含封面、标题、分类、状态
- [ ] “编辑 / 图片 / 发布或撤回” 在单屏内可见
- [ ] 编辑页继续复用现有分类与商品 API，不新增后端字段约定
- [ ] 图片页覆盖上传、设封面、排序、删除四类操作

### 4.5 留言模块

- [ ] 列表可按状态查看，且能进入详情
- [ ] 详情页可回复、标记已读/未读
- [ ] 冷启动直达详情时，如缺少上下文可安全回退到留言列表并提示
- [ ] 回复成功后状态刷新清晰，不出现“已回复但列表仍旧未读”的错觉

### 4.6 桌面回归

- [ ] `/dashboard / products / messages / settings` 等原桌面主路径保持可用
- [ ] 桌面布局未因手机逻辑出现错位、标题异常或守卫回跳异常
- [ ] 现有桌面 API 调用和登录流程未被 `/m` 分流逻辑破坏

## 5. 建议手工回归矩阵

### 5.1 入口分流

| 场景 | 预期 |
| --- | --- |
| 手机 UA + 未登录 + `/` | `/m/login` |
| 手机 UA + 未登录 + `/login` | `/m/login` |
| 桌面 UA + 未登录 + `/` | `/login` |
| 桌面 UA + 已登录 + `/login` | `/dashboard` |
| 手机 UA + 已登录 + `/login` | `/m/home` |

### 5.2 显式路径保持

| 场景 | 预期 |
| --- | --- |
| 手机 UA 打开 `/products` | 保持桌面路径 |
| 桌面 UA 打开 `/m/products` | 保持手机路径 |
| 显式打开 `/m/login` | 保持手机登录页 |

### 5.3 隐藏前缀

| 场景 | 预期 |
| --- | --- |
| `/gate/login` | 可进入桌面登录 |
| `/gate/m/login` | 可进入手机登录 |
| `/gate/products` | 登录后桌面可用 |
| `/gate/m/products` | 登录后手机可用 |

### 5.4 401 恢复

| 场景 | 预期 |
| --- | --- |
| 手机路由下 token 失效 | `/m/login?redirect=...` |
| 桌面路由下 token 失效 | `/login?redirect=...` |
| 重新登录成功 | 回到原 `redirect` 页面 |

### 5.5 业务能力

- [ ] 手机首页能看到核心摘要与快捷入口
- [ ] 商品列表可搜索、筛选、跳转编辑/图片页
- [ ] 商品编辑可加载已有数据并保存
- [ ] 图片管理可上传、设封面、排序、删除
- [ ] 留言列表可筛选并进入详情
- [ ] 留言详情可回复并切换已读/未读

## 6. 静态验证与交付要求

当前仓库前端已具备 TypeScript 与构建脚本，但未看到现成自动化测试框架；一期建议最低交付证据如下：

1. `frontend` 构建通过
2. TypeScript 无阻塞错误
3. 至少完成一轮 Android Chrome / iPhone Safari 手工回归
4. 完成桌面主路径回归检查

建议命令：

```bash
cd frontend
npm exec tsc --noEmit
npm run build
```

如后续补自动化，优先补以下高价值用例：

1. 体验判定与 override 解析
2. 隐藏前缀 + `/m` 路由矩阵
3. 401 回跳路由族保持
4. 留言详情冷启动降级

## 7. 当前实现风险提示

结合当前仓库的桌面后台基线，手机浏览器后台一期最容易出错的点有四个：

1. **把 `/m` 当作 history base 使用**：会直接破坏隐藏前缀部署。
2. **在所有路由上做 UA 自动切换**：会导致显式深链行为不稳定。
3. **401 只回到单一路径 `/login`**：会让手机会话恢复丢失路由族。
4. **过度复用桌面组件**：会让手机端继续保留宽表格/高密度布局问题。

## 8. 一句话交付标准

一期上线前，必须同时满足三件事：

- 中性入口能正确分流到桌面或手机后台
- `/m/...` 手机高频运营路径形成闭环
- 桌面后台现有主路径无明显回归
