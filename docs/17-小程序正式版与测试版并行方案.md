# 小程序正式版与测试版并行方案

版本：v0.2  
日期：2026-04-11

本文用于记录当前项目在“小程序已上线”的前提下，如何同时维持正式版稳定运行，并支持后续接口改动、开发联调、体验版验证与审核切换。

## 1. 当前现状

结合当前仓库代码与部署配置，现状如下：

- 后端通过统一前缀 `/api` 对外提供接口，暂未区分 `v1`、`v2`
- 小程序端通过 `API_ORIGIN + API_PREFIX` 组合接口地址
- 当前 Docker Compose 部署使用 SQLite，本地上传文件也由后端本地目录提供
- 当前后端数据目录挂载到一个持久化卷中，数据库文件与上传文件都在同一个卷内

当前相关实现位置：

- `docker-compose.yml`
- `backend/src/backend/core/config.py`
- `backend/src/backend/main.py`
- `backend/src/backend/api/router.py`
- `miniapp/utils/config.js`
- `miniapp/utils/config.private.example.js`

## 2. 目标

后续方案需要同时满足以下目标：

1. 正式版小程序持续可用，不受开发测试影响
2. 开发版、体验版可以提前联调和验证新接口
3. 接口调整时，尽量避免因破坏性变更导致正式版故障
4. 审核期间可以切换到一套风险更低、敏感度更低的数据
5. 后续如果接口演进复杂，再平滑引入 API 版本号

## 3. 总体建议

当前阶段不建议立刻全面引入 API 版本号。

优先级建议如下：

1. 先明确“短期过渡方案”和“长期标准方案”的边界
2. 短期优先建立可落地的测试流量入口
3. 再逐步完成正式环境与测试环境的全面隔离
4. 建立“向后兼容优先”的接口变更规则
5. 只有在出现明确破坏性升级需求时，再引入 `/api/v2`

原因：

- 当前项目规模仍可控，客户端也主要由自己维护
- 小程序接口地址已经可配置，先做环境隔离成本最低
- 只要接口改动遵守兼容原则，正式版与体验版可以并行一段时间
- 过早全面版本化会增加维护成本、路由复杂度和测试成本

一句话结论：

- 现在最该做的是先让“正式流量”和“测试流量”可并行
- 短期可先走同域名 + `/dev-api` 的过渡方案
- 长期再演进到正式域名与测试域名彻底分离
- 不是立刻做“全量接口版本治理”

## 4. 推荐部署拓扑

本文将部署方案明确分成两层：

- **短期过渡方案**：复用现有正式域名入口，在同一域名下通过路径前缀分流正式流量与测试流量，目标是尽快落地、尽快验证
- **长期标准方案**：正式环境与测试环境在域名、容器、数据、配置上全面隔离，目标是降低串环境风险、提升可维护性

下面的“正式环境 / 测试环境”描述，默认指的是**长期标准方案**；如果当前只是想快速落地测试流量入口，应优先参考后文的“同域名 + 前缀分流”过渡方案。

### 4.1 正式环境

用途：

- 服务正式版小程序用户
- 服务正式管理后台
- 只允许经过验证的变更进入

建议配置：

- API 域名：`https://fashion.dadaye.online/api`
- 后台域名：正式后台域名
- Docker Compose 项目标识：`yrfashion-prod`
- 数据卷：独立正式卷，例如 `backend_data_prod`
- 数据内容：正式 SQLite 数据库 + 正式上传文件

### 4.2 测试环境

用途：

- 服务本地开发联调
- 服务开发版、体验版小程序
- 用于新接口、新页面、新配置验证
- 必要时用于审核数据切换预演

建议配置：

- API 域名：`https://fashion-staging.dadaye.online/api`
- 后台域名：测试后台域名
- Docker Compose 项目标识：`yrfashion-staging`
- 数据卷：独立测试卷，例如 `backend_data_staging`
- 数据内容：脱敏或测试专用 SQLite 数据库 + 测试上传文件

### 4.3 当前线上实际链路（2026-04-11 补充）

结合当前 `docker-compose.yml` 与 `frontend/nginx/default.conf.template`，当前线上链路已经明确为：

`fashion.dadaye.online`
→ DNS 解析到公网 IP `223.22.11.101`
→ 公网机内网 IP `172.26.100.22` 上的 Nginx Proxy Manager
→ 反向代理到业务机 `172.26.100.175:8080`
→ 进入 `frontend` 容器内的 Nginx `80` 端口
→
- 管理后台与静态资源：由 `frontend` 容器直接提供
- `/api/`、`/uploads/`：由 `frontend` 容器内 Nginx 再反向代理到 `backend:8000`

当前 Compose 关键点：

- `frontend` 端口映射：`8080:80`
- `backend` 端口映射：`8000:8000`
- `frontend` 环境变量：`BACKEND_UPSTREAM=backend:8000`

这说明当前 `172.26.100.175:8080` 实际上是“前端入口”，而不是后端直出入口。外部所有来自 `fashion.dadaye.online` 的请求，都会先经过前端容器 Nginx，再决定是回源静态页面，还是转发给后端。

### 4.4 方案分层说明

结合当前讨论，推荐把“双环境并行”拆成两个层次理解：

#### 第一层：短期过渡方案

适用场景：

- 还没有准备好独立测试域名
- 想尽快让开发版 / 体验版小程序接入测试接口
- 希望尽量少改公网入口与微信合法域名配置

推荐形态：

- 正式版小程序：`https://fashion.dadaye.online/api`
- 测试版小程序：`https://fashion.dadaye.online/dev-api`
- 同一个公网域名入口
- 同一个前端容器 Nginx 负责路径分流
- `/api` 指向正式后端，`/dev-api` 指向测试后端

这一层的重点是：**先把测试流量和正式流量分开走不同后端**，而不是一步到位把所有基础设施都拆开。

#### 第二层：长期标准方案

适用场景：

- 测试链路已经稳定，需要降低串环境风险
- 需要更清晰的运维边界、备份策略、权限边界
- 后续要长期维护开发版 / 体验版 / 审核版小程序

推荐形态：

- 正式环境使用正式域名
- 测试环境使用独立测试域名
- 两套环境使用独立 Compose 项目、独立容器、独立数据卷、独立配置

这一层的重点是：**不仅分流量，还要分基础设施与数据边界**。

### 4.5 长期独立环境建议隔离的内容

当项目进入“长期标准方案”时，正式环境与测试环境建议至少在以下维度完全隔离：

- Docker Compose 项目名
- 容器
- 端口
- 持久化卷
- 域名
- `SECRET_KEY`
- 管理员账号密码
- 小程序请求域名白名单

不建议共用以下内容：

- 同一个 `backend_data` 卷
- 同一个 SQLite 文件
- 同一套上传目录
- 同一个后台登录凭据

## 5. 数据策略

当前 Compose 配置中，后端将 `/app/data` 挂载到持久化卷，该目录至少包含：

- `app.db`
- `uploads/`

因此对于当前 SQLite 部署，环境切换和备份恢复都应以“整套数据目录”视角处理，而不是只盯数据库文件。

### 5.1 正式环境数据

正式环境数据应保持完整，作为真实线上数据。

建议：

- 按周期备份正式环境 `backend_data_prod`
- 备份时同时覆盖数据库文件和上传文件
- 进行备份时优先停后端容器，避免 SQLite 热拷贝不一致

### 5.2 测试环境数据

测试环境建议使用以下两种方式之一：

1. 长期维护一套固定的测试数据
2. 从正式环境周期性复制后再做脱敏处理

推荐第二种，但必须脱敏后再用于测试或审核。

建议脱敏范围：

- 用户昵称
- 用户头像
- 留言内容
- 联系方式
- 任何可能识别真实客户的信息

### 5.3 审核数据切换

如果后续小程序审核需要一套更稳妥的数据，建议不是直接改正式环境，而是：

1. 预先准备一套审核专用测试卷
2. 开发版或体验版先接入审核数据环境验证
3. 审核用代码包指向指定测试环境
4. 正式版继续使用正式环境，不受影响

如果确实需要临时替换某套环境数据，也建议切换整套 `backend_data` 快照，而不是只替换 `app.db`。

## 6. 小程序多环境策略

当前小程序配置已经支持通过 `miniapp/utils/config.js` 指定接口地址，因此后续只需要明确“哪个包连接哪个环境”。

建议约定如下：

- 本地开发：连接测试环境
- 微信开发版：连接测试环境
- 微信体验版：连接测试环境
- 微信正式版：连接正式环境

这里也建议明确分成两层配置：

- **短期过渡方案**：同一个 `API_ORIGIN`，通过不同 `API_PREFIX` 区分正式与测试
- **长期标准方案**：正式与测试使用不同 `API_ORIGIN`，`API_PREFIX` 仍统一保持 `/api`

### 6.1 短期过渡方案：同域名 + 前缀分流

在当前部署形态下，还可以采用一个过渡方案：继续复用正式域名 `fashion.dadaye.online`，但通过前缀区分正式接口与测试接口。

示例：

- 正式版小程序：`https://fashion.dadaye.online/api`
- 测试版小程序：`https://fashion.dadaye.online/dev-api`

这个方案可行的原因是：当前所有请求都会先进入 `frontend` 容器内 Nginx，因此可以直接在 `frontend/nginx/default.conf.template` 里增加额外路由，例如：

- `/api/` → 正式后端
- `/dev-api/` → 测试后端

推荐做法：

1. 在 Nginx 中新增 `location /dev-api/`
2. 将 `/dev-api/...` 重写为后端可识别的 `/api/...` 路径
3. 引入独立的 `DEV_BACKEND_UPSTREAM`，让 `/dev-api/` 转发到测试环境后端

例如目标可以是：

- `BACKEND_UPSTREAM=backend:8000`
- `DEV_BACKEND_UPSTREAM=backend-dev:8000`

或指向另一台测试机器。

对应到小程序配置，可以理解为：

- 正式版：
  - `API_ORIGIN=https://fashion.dadaye.online`
  - `API_PREFIX=/api`
- 测试版：
  - `API_ORIGIN=https://fashion.dadaye.online`
  - `API_PREFIX=/dev-api`

这个方案的优点：

- 不需要新增请求域名，适合微信小程序当前已接入的同一合法域名
- Nginx Proxy Manager 无需额外加一条新域名入口
- 前端容器本身就已经承担统一反向代理职责，改造成本较低

这个方案的限制：

- 它只是“同域名下的路径级分流”，不是完整的域名级隔离
- 如果 `/dev-api` 最终仍指向正式后端，那只是在换前缀，不算真正测试环境
- 长期来看，独立测试域名仍然比路径前缀方案更清晰、更利于排查与权限管理

因此更建议把它定义为：

- **短中期可落地的过渡方案**：同域名下增加 `/dev-api`
- **长期标准方案**：正式域名与测试域名彻底分离

### 6.2 长期标准方案：独立测试域名

当测试环境准备完善后，再演进到更标准的地址规划：

- 正式：
  - `API_ORIGIN=https://fashion.dadaye.online`
  - `API_PREFIX=/api`
- 测试：
  - `API_ORIGIN=https://fashion-staging.dadaye.online`
  - `API_PREFIX=/api`

此时需要在微信小程序后台配置请求合法域名：

- 正式接口域名
- 测试接口域名

否则开发版或体验版无法正常请求测试环境。

### 6.3 发布约定

建议建立以下发布约定：

1. 所有新接口先部署到测试环境
2. 开发版和体验版先验证新接口
3. 验证通过后，再部署到正式环境
4. 正式版代码包只连接正式环境

这样可以避免“一个未验证的后端改动直接影响线上正式版”。

## 7. 接口演进策略

### 7.1 当前阶段的原则

当前阶段以“向后兼容优先”为主，不急于上版本号。

建议的接口变更规则：

- 可以新增接口
- 可以新增响应字段
- 可以新增可选请求参数
- 不要直接删除旧接口
- 不要直接删除旧字段
- 不要在不兼容的前提下修改字段含义

数据库结构变更也尽量遵守类似原则：

- 优先新增列，而不是直接替换旧列
- 优先让旧代码仍可运行
- 等小程序新版本稳定覆盖后，再清理旧逻辑

### 7.2 破坏性变更如何处理

如果后续出现以下情况，就要考虑引入 API 版本号：

- 接口路径要整体重组
- 响应结构要大幅调整
- 旧字段必须删除
- 旧客户端无法兼容新协议
- 正式版与新体验版需要长期并行不同接口协议

此时建议只优先给小程序公开接口加版本号，例如：

- `/api/v1/miniapp/...`
- `/api/v2/miniapp/...`

后台管理接口可以暂时继续保持：

- `/api/admin/...`

这样可以把版本治理控制在最需要的边界内，避免一次性改全项目。

### 7.3 为什么现在不建议立刻加版本号

因为当前你真正要解决的问题是：

- 正式版不能被测试接口影响
- 体验版要能提前验证
- 审核期要能切到更合适的数据

这些问题本质上都属于“环境隔离”问题，不是“路由版本号”问题。

如果环境没有隔离，即使加了 `/api/v1`，你仍然可能：

- 用同一套数据库
- 用同一套上传文件
- 让未验证改动进入线上服务

所以先做版本号，并不能直接解决你眼前的主要风险。

## 8. 推荐的变更流程

后续每次小程序接口改动，建议采用以下流程：

1. 在测试环境部署后端变更
2. 开发版小程序连接测试环境联调
3. 体验版小程序连接测试环境做业务验证
4. 若接口保持兼容，则再把后端变更部署到正式环境
5. 提交新的小程序版本审核
6. 新版本正式发布并稳定运行一段时间后，再考虑删除旧兼容逻辑

如果是破坏性改动，则流程调整为：

1. 在测试环境新增 `/api/v2`
2. 新小程序版本改连 `v2`
3. 正式版仍继续使用 `v1`
4. 待正式版覆盖完成后，再下线 `v1`

## 9. 对当前项目的实施建议

为了尽快落地，后续实施建议按以下顺序推进：

1. 先把“短期过渡方案”和“长期标准方案”作为两个不同阶段固定下来，避免混用概念
2. 如需快速落地，先在现有单域名入口上补 `/dev-api` 前缀分流
3. 为 `/dev-api` 准备一套独立测试后端，确保它不是指回正式环境
4. 让开发版 / 体验版小程序先接入 `/dev-api` 完成联调
5. 然后把 Docker Compose 拆成正式环境与测试环境两套配置
6. 为两套环境分配独立卷，例如 `backend_data_prod` 与 `backend_data_staging`
7. 再补一套测试环境域名，逐步从“路径分流”演进到“域名分流”
8. 形成正式环境与测试环境的备份、恢复、切换操作手册
9. 再视接口演进复杂度决定是否追加 `/api/v2`

## 10. 后续实施项

本文先记录方案，不直接修改当前部署。

后续可继续拆分为以下实施文档或任务：

- Docker Compose 双环境目录与配置改造
- 正式环境与测试环境域名规划
- `backend_data` 备份与恢复脚本
- 测试数据脱敏方案
- 小程序正式/测试环境切换方式
- 小程序接口 `v2` 设计与迁移方案

## 11. 结论

当前项目最合适的路线是：

- 先做正式环境与测试环境双环境隔离
- 保持接口变更尽量向后兼容
- 暂不急于全面引入 API 版本号
- 只在出现明确破坏性升级需求时，再对小程序公开接口引入 `/api/v2`

这条路线更符合你当前的项目阶段，也更能控制小程序正式版上线后的实际风险。
