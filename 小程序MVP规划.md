# 小程序 MVP 规划

版本：v0.1  
日期：2026-03-13

## 1. 项目背景

当前业务以线下服装店经营为主，希望通过微信小程序补充线上展示与客户触达能力。

现阶段目标不是立即做完整电商闭环，而是先完成一个可上线、可运营、可持续迭代的第一版产品，用于：

- 展示商品
- 吸引客户浏览和咨询
- 收集用户留言与反馈
- 让店主方便地维护商品内容

## 2. 产品定位

第一版产品定位为：

`服装展示型微信小程序 + 管理后台`

核心作用：

- 作为线上商品橱窗
- 作为客户咨询入口
- 作为品牌宣传和老客户沉淀工具

第一版暂不以成交闭环为核心目标，因此不优先上线支付、购物车、订单等电商能力。

## 3. 目标用户

### 3.1 外部用户

- 到店老客户
- 通过微信传播进入小程序的新客户
- 对店内服装风格感兴趣的潜在客户

### 3.2 内部用户

- 店主本人
- 后续可能协助运营商品的店员

## 4. 第一版核心目标

第一版需要解决的核心问题：

- 用户能方便浏览服装图片和描述
- 用户能对商品留言、咨询
- 店主能方便上传和维护商品内容
- 店主能及时看到新留言并处理

## 5. MVP 功能范围

### 5.1 小程序端

建议第一版包含以下页面与能力：

1. 首页
- 展示店铺介绍、品牌风格、推荐商品入口

2. 商品列表页
- 按后台配置顺序展示商品
- 支持简单分类或标签展示

3. 商品详情页
- 展示多张图片
- 展示文字描述
- 支持留言/咨询

4. 留言/咨询能力
- 用户可提交留言
- 店主可后续回复

5. 联系方式页
- 展示微信、电话、门店地址、营业时间等

### 5.2 管理后台

建议第一版包含以下能力：

1. 登录管理
- 店主后台账号登录

2. 商品管理
- 新增商品
- 编辑商品
- 上下架商品
- 上传商品图片
- 编辑商品描述

3. 商品排序
- 调整展示顺序

4. 留言管理
- 查看留言列表
- 标记已读/未读
- 回复留言

5. 用户列表
- 记录小程序登录用户的基础标识信息
- 第一版不把“完整微信资料”作为核心依赖

## 6. 第一版不做的内容

以下内容建议明确排除在 MVP 之外：

- 在线支付
- 购物车
- 订单系统
- 优惠券
- 复杂会员体系
- 多角色权限系统
- iPad 原生 App

这样可以把第一版重点集中在“展示”和“内容运营”上，降低开发复杂度和上线风险。

## 7. 管理端形态建议

建议第一版采用：

`响应式 Web 管理后台`

理由：

- 电脑端使用方便，适合录入和管理
- iPad 的 Safari 浏览器也可直接访问
- 开发成本明显低于原生 iPad App
- 后续如果确实有需要，再考虑包装为 PWA 或独立 App

当前不建议第一版直接开发 iPad 原生应用，原因如下：

- 技术门槛更高
- 发布和分发流程更复杂
- 维护成本高
- 当前业务场景并不要求原生能力

## 8. 技术方案建议

## 8.1 推荐总体架构

建议优先考虑以下两种方案：

### 方案 A：小程序 + 自建后端 + Web 管理后台

- 前端：微信小程序
- 管理端：Vue 或 React 的响应式 Web 后台
- 后端：Java Spring Boot 或 Python FastAPI
- 数据库：MySQL
- 图片存储：对象存储或云存储

适合情况：

- 希望后续扩展能力更强
- 希望技术栈更可控
- 后续可能继续加入订单、支付、会员等能力

### 方案 B：小程序 + 云开发 / 云托管

- 前端：微信小程序
- 管理端：Web 后台
- 后端：微信云开发或腾讯云 CloudBase/云托管
- 存储：云存储
- 数据：云数据库或托管数据库

适合情况：

- 第一版更看重快速上线
- 想减少服务器和运维投入
- 团队规模较小，优先验证业务

## 8.2 后端语言建议

如果更熟悉 Java，建议优先使用：

`Java + Spring Boot`

理由：

- 生态成熟
- 管理后台和业务接口都容易规范化
- 后续扩展到订单和支付也更顺

如果更想快速验证，也可以使用：

`Python + FastAPI`

选择原则不是“哪个微信 SDK 更多”，而是：

- 哪个技术栈你更熟
- 哪个技术栈能让第一版更快稳定上线

## 9. 数据与内容设计建议

第一版建议围绕以下核心对象设计：

### 9.1 商品

- 商品名称
- 商品封面图
- 商品图片集
- 商品描述
- 分类/标签
- 展示顺序
- 上架状态
- 创建时间

### 9.2 用户

- openid
- 昵称（如用户授权后获取）
- 头像（如用户授权后获取）
- 首次访问时间
- 最近访问时间

说明：

用户信息设计不应依赖过多微信侧资料，核心应以业务可用性为准。

### 9.3 留言

- 用户标识
- 对应商品
- 留言内容
- 留言时间
- 回复内容
- 回复时间
- 状态（未读、已读、已回复）

## 10. 留言提醒方案

留言提醒建议优先在服务端和管理端实现，不建议把提醒逻辑放在消费者使用的小程序前台中。

第一版建议分层处理：

1. 基础版
- 后台显示未读留言数
- 后台留言列表按未处理优先展示

2. 增强版
- 邮件提醒
- 企业微信提醒
- 短信提醒

3. 后续版
- 基于微信订阅消息做特定通知能力

第一版先做后台未读提醒即可，足以支持日常运营。

## 11. 上线前的关键现实条件

以下内容在正式上线前必须提前考虑：

### 11.1 小程序账号

需要注册微信小程序账号。小程序是独立账号类型，不必先开通订阅号或服务号。

### 11.2 登录机制

微信小程序登录流程的核心是：

- 小程序端调用 `wx.login()`
- 服务端通过 `code2Session` 换取 `openid` 等标识

因此需要有后端服务参与登录态建立。

### 11.3 网络与域名

正式环境下，小程序请求后端通常需要：

- HTTPS
- 已配置的服务器域名
- 合规证书

开发阶段可临时调试，但正式版不建议依赖“本地电脑 + 内网穿透”长期运行。

### 11.4 备案

正式发布前要关注小程序备案与相关合规要求。备案不建议拖到最后才处理，应尽早纳入项目计划。

## 12. 服务器与成本判断

如果第一版只是展示型小程序，且用户量不大，建议配置从轻量方案起步。

推荐原则：

- 应用服务轻量化
- 图片不要存本机磁盘，改用对象存储或云存储
- 能用托管服务的地方尽量少自运维

初步判断：

`1C2G` 对第一版通常足够

前提：

- 图片走对象存储
- 业务接口不复杂
- 并发量不高

## 13. 分阶段实施建议

### 阶段一：产品确认

- 明确页面范围
- 明确后台功能
- 明确商品字段
- 明确留言流程

### 阶段二：基础搭建

- 注册小程序账号
- 确定技术栈
- 搭建项目骨架
- 确定部署方案

### 阶段三：MVP 开发

- 完成小程序首页、列表、详情、留言
- 完成后台商品管理、排序、留言管理
- 完成图片上传

### 阶段四：测试与上线准备

- 真机测试
- 图片与加载速度优化
- 备案与域名准备
- 提交审核上线

### 阶段五：二期迭代

- 收藏/预约
- 营销活动
- 订阅通知
- 支付与订单

## 14. 当前推荐结论

当前最推荐的起步方案如下：

- 产品方向：先做展示型小程序
- 管理端：响应式 Web 后台
- 技术路线：优先选你最熟的后端技术栈
- 部署方式：优先考虑云开发、云托管或轻量云服务器
- 范围控制：第一版不做支付和订单
